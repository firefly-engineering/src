# Firefly Engineering Monorepo Architecture

## Overview

This monorepo combines **Nix** for environment and toolchain management with **Buck2** as the build system. The two are bridged by [turnkey](https://github.com/firefly-engineering/turnkey), which generates Buck2 configuration from a single declarative source so that native tooling and Buck2 builds always reference the exact same binaries.

The supporting libraries are:

- **[nix-pins](https://github.com/firefly-engineering/nix-pins)** — centralized flake input pins shared across all Firefly repos.
- **[teller](https://github.com/firefly-engineering/teller)** — versioned toolchain registry library (no toolchain versions, just the mechanism).
- **[toolbox](https://github.com/firefly-engineering/toolbox)** — registry of curated toolchain and dependency packages built from upstream releases.
- **[turnkey](https://github.com/firefly-engineering/turnkey)** — flake-parts and devenv modules that wire `toolchain.toml` declarations into a dev shell, Buck2 toolchains, and generated dependency cells.

## Design Principles

1. **Single source of truth.** `toolchain.toml` declares what versions are wanted; turnkey resolves them through teller + toolbox into Nix derivations used by both the shell and Buck2.
2. **Hermetic and reproducible.** All inputs are flake-pinned and content-addressed. Identical inputs produce identical outputs.
3. **Non-contaminating.** Standard language tooling (`cargo`, `go build`, IDE language servers) works unmodified inside the shell.
4. **Minimal vendor lock-in.** Projects can be extracted with their standard manifests (`Cargo.toml`, `go.mod`, etc.) intact.

## Components

### Flake inputs

The root `flake.nix` pulls everything through `nix-pins` so that nixpkgs, flake-parts, and devenv resolve to the same revisions as the rest of the ecosystem. Teller and toolbox come in transitively via turnkey and don't need to be declared at the top level:

```nix
inputs = {
  nix-pins.url = "github:firefly-engineering/nix-pins";
  nixpkgs.follows = "nix-pins/nixpkgs";
  flake-parts.follows = "nix-pins/flake-parts";

  turnkey = {
    url = "github:firefly-engineering/turnkey";
    inputs.nix-pins.follows = "nix-pins";
  };

  devenv.follows = "turnkey/toolbox/devenv";
};
```

### Toolchain declaration

`toolchain.toml` is the only file developers edit to change what's in the shell:

```toml
[toolchains]
buck2 = {}
tk = {}
go = {}
rust = {}
cargo = {}
python = {}
nodejs = {}
typescript = {}
nix = {}
jj = {}
beadwork = {}
mdbook-toolchain = {}
```

Each entry resolves to a versioned package from toolbox's registry. The empty table means "use the registry default"; specific versions can be pinned per-toolchain.

### Buck2 cells (turnkey-managed)

On shell entry, turnkey writes `.buckconfig` as a symlink to a Nix-generated config and creates symlinks under `.turnkey/` for the cells it manages:

```
.buckconfig                 → /nix/store/.../turnkey.buckconfig  (symlink)
.buckroot                   (marks project boundary)
.turnkey/
├── prelude/                → Nix-managed Buck2 prelude
├── toolchains/             → Generated system toolchains (go, rust, etc.)
├── godeps/                 → Generated cell for Go dependencies
└── rustdeps/               → Generated cell for Rust dependencies
```

The root cell uses `rules.star` as its buildfile name (set by turnkey in `.buckconfig`). Cells under `.turnkey/` use `BUCK` — Buck2's `[buildfile]` setting only applies to the root cell, so generated cells keep the default.

### Per-language dependency declarations

Each managed language has a single deps file that drives generation of the matching `.turnkey/*deps/` cell:

- `go-deps.toml` — generated by `godeps-gen` from `go.mod` + `go.sum`.
- `rust-deps.toml` — generated by `rustdeps-gen` from `Cargo.lock`.

Buck2 references these as cells (e.g. `godeps//vendor/golang.org/x/example/hello/reverse:reverse`). Native tooling continues to consume `go.mod` / `Cargo.toml` directly — the deps files are just a Buck2-side view of the same dependency graph.

### Configuration entry point

`nix/shell.nix` wires it all up via the turnkey flake-parts module. `tellerLib` and `tellerRegistry` default to the bundled teller + toolbox setup, so the consumer only declares the project-specific bits:

```nix
turnkey.toolchains = {
  enable = true;
  declarationFiles.default = ../toolchain.toml;

  buck2 = {
    enable = true;
    go.depsFile   = ../go-deps.toml;
    rust.depsFile = ../rust-deps.toml;
  };
};
```

Override `tellerLib`/`tellerRegistry` only when targeting a non-default registry (e.g. a private overlay on top of toolbox).

## Build files

Root-cell build files live as `rules.star` next to the code they describe:

```
experimental/
├── rs-hello-world/
│   ├── rules.star          # rust_binary target
│   ├── Cargo.toml
│   └── src/main.rs
└── go-hello-world/
    ├── rules.star          # go_binary target
    ├── go.mod
    └── main.go

docs/
├── rules.star              # mdbook_book target
├── book.toml
└── src/**.md
```

A target looks like:

```python
# experimental/go-reverse-hello-world/rules.star
go_binary(
    name = "go-hello-world",
    srcs = ["main.go"],
    deps = [
        "godeps//vendor/golang.org/x/example/hello/reverse:reverse",
    ],
    visibility = ["PUBLIC"],
)
```

## Development workflow

1. **Enter the shell.** `direnv allow` (or `nix develop`) triggers turnkey's `enterShell` hook. It materializes `.turnkey/` cells, points `.buckconfig` at the generated config, and regenerates `go-deps.toml`/`rust-deps.toml` if `go.mod` or `Cargo.lock` changed.
2. **Build with Buck2.** `buck2 build //path/to:target` or `tk build //path/to:target` (the `tk` wrapper auto-syncs deps files before delegating to `buck2`).
3. **Use native tooling.** `cargo build`, `go test`, IDE language servers — all see the same toolchain binaries that Buck2 sees because both contexts resolve to the same Nix store paths.

## Why this matters

- **No drift.** Shell `go version` and Buck2 `go_binary` always invoke the same `/nix/store/.../go` derivation.
- **Cache-friendly.** Buck2 cache keys include Nix store paths, which only change when the input derivation changes. Identical inputs across machines produce identical cache hits.
- **Replaceable parts.** Turnkey is the mechanism, toolbox is the data. Either can be swapped or extended (e.g. a private registry overlay) without touching the other.
