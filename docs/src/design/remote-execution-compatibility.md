# Nix-Backed Toolchains for Remote Execution in Buck2: The Manifest & Archive Strategy

When integrating Nix-managed toolchains with Buck2's Remote Execution (RE) capabilities, the goal is to leverage Nix's hermetic package management while ensuring Buck2 can accurately identify and transfer *all* necessary toolchain files to the remote worker. This document outlines a robust strategy: the "Manifest and Archive" pattern, which provides explicit control over the toolchain's contents and ensures complete hermeticity.

## The Challenge: Bridging Nix and Remote Execution

The core challenge lies in the different mental models:

  * **Nix:** Builds derivations and places their outputs in the `/nix/store`, relying on the Nix daemon to manage the transitive closure.
  * **Buck2 Remote Execution:** Operates in a strict sandbox. Every file needed by an action *must* be explicitly declared as an input so Buck2 can upload it to the Content-Addressable Store (CAS) and provision the sandbox. The remote worker does **not** have a Nix daemon or `/nix/store`.

The naive approach (pointing Buck2 to a `/nix/store/...` path) is problematic because:

1.  It relies on the host's Nix installation.
2.  Buck2 needs to know *all* files within that path, not just the root.

The "Manifest and Archive" strategy addresses these by creating a fully self-contained, Buck2-managed toolchain artifact.

## Strategy: Manifest & Archive Pattern

This strategy involves a series of Buck2 rules that progressively transform Nix's output into a remote execution-compatible input.

### Step 1: Define Your Nix Toolchain (Nix Flake)

First, ensure your toolchain is defined reproducibly in Nix, ideally using a Nix flake. This allows you to pin all dependencies.

**`flake.nix` (example for a C++ toolchain):**

```nix
{
  description = "A C++ toolchain for Buck2";

  inputs = {
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-23.11";
  };

  outputs = { self, nixpkgs }: {
    devShells.default = { pkgs }: pkgs.mkShell {
      packages = with pkgs; [
        gcc12 # Your desired C++ compiler
        gnumake
      ];
    };
    packages.default = let pkgs = import nixpkgs { system = "x86_64-linux"; }; in pkgs.gcc12; # The toolchain itself
  };
}
```

**`flake.lock`:** This file (generated by `nix flake update` and committed to your repo) explicitly pins all Nix inputs, making your toolchain definition hermetic.

### Step 2: Generate the Toolchain Manifest (Buck2 `genrule`)

This rule runs *locally* and queries Nix to get the complete transitive closure of your toolchain.

**`toolchains/BUCK`:**

```python
# Rule to get the Nix store path and its full transitive closure
genrule(
    name = "toolchain_manifest",
    srcs = [
        "//:flake.nix",      # Depend on your flake
        "//:flake.lock",     # Depend on your pinned lock file
    ],
    # The command should query Nix for the toolchain's store path and then for the full closure.
    cmd = """
        set -euo pipefail
        # Get the primary store path of the toolchain
        FLAKE_OUT_PATH=$$(nix build -L --print-out-paths $(location //:flake.nix)#packages.$(uname -m)-linux.default)

        # Output the primary store path to a temporary file
        echo "$$FLAKE_OUT_PATH" > $OUT_DIR/primary_path.txt

        # Use nix path-info to list all transitive dependencies of the primary path
        # And write them into the manifest file
        nix path-info --json --references-graphs --closure --all "$$FLAKE_OUT_PATH" | \
            jq -r '.[] | .path' | \
            sort > $OUT
    """,
    out = "manifest.txt", # This file lists all `/nix/store/...` paths in the closure
    # Crucially, this rule must run locally, not on RE, as it interacts with the local Nix daemon.
    # Remote execution for this rule would fail without a Nix daemon in the sandbox.
    labels = ["no_remote_execution"], # Prevent sending this to RE
)
```

  * **`manifest.txt` output:** This file will contain a newline-separated list of absolute `/nix/store/...` paths for every component of your toolchain.

### Step 3: Create a Hermetic Toolchain Archive (Buck2 `genrule`)

This rule takes the `manifest.txt` and bundles all those `/nix/store/...` paths into a single `.tar` archive.

**`toolchains/BUCK` (continued):**

```python
# Rule to archive all files specified in the manifest.
genrule(
    name = "toolchain_archive",
    srcs = [":toolchain_manifest"], # Depends on the manifest
    cmd = """
        set -euo pipefail
        # The manifest lists absolute paths. We want to archive them relative to `/nix/store` to keep paths sane.
        # So we create a temporary directory and link/copy all paths into it, preserving their structure relative to /nix/store.
        TMP_ARCHIVE_DIR=$(mktemp -d)
        trap "rm -rf $$TMP_ARCHIVE_DIR" EXIT # Clean up temp dir on exit

        # Read manifest, create directories and link files into the temp archive dir
        while IFS= read -r store_path; do
            # The toolchain may contain very long paths; handle gracefully
            relative_path="${store_path#/nix/store/}"
            mkdir -p "$$TMP_ARCHIVE_DIR/nix/store/$(dirname "$$relative_path")"
            ln -s "$$store_path" "$$TMP_ARCHIVE_DIR/nix/store/$$relative_path"
        done < "$(location :toolchain_manifest)"

        # Create the tar archive from the temporary directory
        tar -C "$$TMP_ARCHIVE_DIR" -cvf $OUT .
    """,
    out = "toolchain.tar", # This is your self-contained toolchain artifact
    labels = ["no_remote_execution"], # Also runs locally as it depends on local Nix paths
)
```

  * **`toolchain.tar` output:** This archive contains all the toolchain files, structured starting from `/nix/store/`. Its content hash is completely determined by your `flake.lock`.

### Step 4: Define a Starlark `toolchain` Rule

Now, you create a custom Starlark rule that "exposes" this archived toolchain to other Buck2 rules.

**`toolchains/toolchain_defs.bzl`:**

```python
load("@prelude//toolchains:toolchain.bzl", "toolchain_type")
load("@prelude//rust:toolchain_types.bzl", "RustToolchainInfo") # Example for Rust, adapt for C++/Go

# Define a toolchain type for your Nix-managed C++ toolchain
nix_cxx_toolchain_type = toolchain_type(
    name = "nix_cxx_toolchain_type",
    # You might define custom providers here if needed
)

# Implementation of the Nix-backed C++ toolchain rule
def _nix_cxx_toolchain_impl(ctx):

    unpacked_dir = ctx.actions.archive.extract(
        src = ctx.attrs.toolchain_archive,
        output_directory = "nix_cxx_toolchain_unpacked",
    )

    # Now, find the actual compiler binary within the unpacked directory.
    # This path is relative to the *root of the archive*.
    # You'll need to know the specific path from your Nix derivation.
    # Example: If `gcc12` is at /nix/store/...-gcc-12.3.0/bin/gcc
    # And your archive is rooted at the /nix/store/ level.
    # You'd find it at: unpacked_dir.resolve("nix/store/<hash>-gcc-12.3.0/bin/gcc")
    
    # A more robust way might be to inspect the contents, or for complex
    # toolchains, create a small script in the archive that points to the binaries.
    
    # For simplicity, let's assume you know the toolchain's root store path
    # and the relative path to the binary.
    primary_store_path_file = ctx.actions.read_file(ctx.attrs.primary_store_path_file)
    primary_store_path = primary_store_path_file.text.strip()
    
    # Extract the hash and name from the primary_store_path (e.g., /nix/store/HASH-gcc-VERSION)
    # This might require some string manipulation
    
    # Example: Assuming the main compiler is at `bin/gcc` relative to the *root of the unpacked derivation*
    # You might pass the actual path of the compiler inside the derivation as an attribute to this rule.
    # For now, let's assume the root of the unpacked archive is `/nix/store/<hash>-gcc-version`
    # You might get the specific store path of `gcc` from the manifest itself.

    # Simpler approach: Assume the *unpacked_dir* itself acts as the toolchain root
    # This means your tar archive should have had its contents structured so that
    # e.g., `bin/gcc` is at `nix_cxx_toolchain_unpacked/bin/gcc`
    
    # Let's adjust the tar command slightly to make this easier:
    # Instead of `tar -C "$$TMP_ARCHIVE_DIR" -cvf $OUT .`
    # You could unpack the toolchain into a known directory first,
    # and then archive that *specific* directory's contents.
    # For this example, let's assume a structure like:
    # `unpacked_dir/bin/gcc`
    
    # In practice, you'd pass specific binary paths as attributes, or discover them.
    # For now, let's assume the root of the C++ toolchain within the archive is known.
    # E.g., if you've put `gcc12` at the root of the archive:
    compiler_bin = unpacked_dir.resolve("bin/gcc") # Adjust this path for your actual toolchain

    # Create a FileSet containing *all* files in the unpacked directory.
    # This ensures that Buck2 tracks every single file from the Nix closure.
    all_toolchain_files = ctx.actions.get_files_in_directory(unpacked_dir)

    # Return the ToolchainInfo. Adapt this to the specific toolchain provider you need
    # (e.g., CxxToolchainInfo for C++, GoToolchainInfo for Go, etc.)
    return [
        # Example for a generic ToolchainInfo, adapt to specific language
        DefaultInfo(
            sub_targets = {
                "compiler": [compiler_bin],
                "all_files": all_toolchain_files,
            },
        ),
        # If using C++, you'd return `CxxToolchainInfo`
        # cxx_toolchain_info(
        #     compiler = compiler_bin,
        #     linker = unpacked_dir.resolve("bin/ld"), # Example
        #     all_deps = all_toolchain_files,
        #     # ... other CxxToolchainInfo fields
        # )
    ]

nix_cxx_toolchain = rule(
    impl = _nix_cxx_toolchain_impl,
    attrs = {
        "toolchain_archive": attr.artifact(), # Input is the tar archive
        "primary_store_path_file": attr.source(), # File containing the primary store path
    },
    provides = [nix_cxx_toolchain_type], # Declare what this rule provides
)
```

**`toolchains/BUCK` (add rule instance):**

```python
# Instance of your Nix-backed toolchain rule
load(":toolchain_defs.bzl", "nix_cxx_toolchain")

nix_cxx_toolchain(
    name = "my_nix_cxx_toolchain",
    toolchain_archive = ":toolchain_archive", # Depends on the archive from Step 3
    primary_store_path_file = ":toolchain_manifest[primary_path.txt]", # Pass the file with the primary path
)
```

### Step 5: Use the Toolchain in Your Build Rules

Finally, your language-specific rules (e.g., `cxx_binary`, `go_binary`) can depend on and utilize this Starlark-defined toolchain.

**`src/BUCK`:**

```python
load("//toolchains:toolchain_defs.bzl", "nix_cxx_toolchain_type") # Load the type

cxx_binary(
    name = "my_app",
    srcs = ["main.cpp"],
    # Declare the dependency on your Nix toolchain
    toolchains = [
        "//toolchains:my_nix_cxx_toolchain",
    ],
    # The actual implementation of `cxx_binary` would then access the
    # `nix_cxx_toolchain_type` provider from the dependency to get the compiler,
    # linker, and *all* the associated files.
)
```

### How this achieves RE compatibility and control:

1.  **Hermetic Toolchain Artifact (`toolchain.tar`):** This is the core. It's a single file whose hash is entirely determined by your `flake.lock`. It contains *every* file from the Nix closure.
2.  **Explicit Input Declaration:** The `nix_cxx_toolchain` rule extracts this archive. Crucially, when it provides the `ToolchainInfo`, it declares *all* extracted files as inputs.
3.  **Action Key Inclusion:** When `cxx_binary` (or any other rule) uses this toolchain, all those declared files become part of the build action's inputs. This means their content hashes are incorporated into the Action Key.
4.  **Remote Execution Sandbox:** When a `cxx_binary` action is sent to a remote worker:
      * Buck2 detects all the files it needs (source, compiler, linker, shared libs, headers from the unpacked archive).
      * It uploads them to the CAS if they aren't already there.
      * The remote worker's sandbox is populated with *only* these explicitly declared files. It does not need a Nix daemon or `/nix/store`.
      * Any change in your `flake.lock` (even a tiny update to a transitive dependency) will change the `manifest.txt`, which changes `toolchain.tar`, which changes the inputs to `cxx_binary`, forcing a cache miss and a rebuild/re-upload to the CAS.

This "Manifest and Archive" pattern is the most robust way to integrate Nix with Buck2 RE, giving you full control over the toolchain's contents and ensuring maximum hermeticity and reproducibility.
